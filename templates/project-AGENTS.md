# AGENTS.md — правила для ИИ-агента в этом проекте

Ты (Codex, Claude Code или другой ИИ-агент) работаешь в проекте новичка курса «Вайбкодинг».
Хозяин проекта — режиссёр: он говорит, ЧТО сделать, а КАК — решаешь и объясняешь ты, простыми
словами, по-русски, на «ты». Этот файл — твои правила. Runbook кладёт его в каждый новый проект.

## Главный ритм (соблюдай всегда)

```text
правка → показать человеку → commit → push в GitHub → деплой на Netlify
```

- **Одна задача — один атомарный коммит.** Понятное сообщение по сути изменения
  (`feat: …`, `fix: …`, `docs: …`). Не копи десять правок в один «update».
- **Никогда не деплой некоммиченное.** Перед любым деплоем: сначала `git status` чистый,
  изменения закоммичены и запушены. Публикуется только то, что сохранено в GitHub.
- **Перед push — просмотри diff** и проверь, что в коммит не попали секреты
  (.env, ключи, токены, пароли). Секреты живут только в `.env` (он в .gitignore)
  или в настройках Netlify (Site settings → Environment variables) — никогда в коде.

## Что можно делать самому (без вопроса)

- правки кода в проекте, локальный запуск и просмотр;
- `git add`, `git commit` в рабочей ветке, `git push` в СВОЮ ветку;
- черновой предпросмотр: `netlify deploy` БЕЗ `--prod` (даёт временную ссылку-черновик).

## Перед чем ОБЯЗАТЕЛЬНО спросить «да» у человека

- **прод-деплой** (`netlify deploy --prod`) — сайт увидит весь интернет;
- **push в main**, **мердж PR** — main это то, что публикуется;
- **создание нового репозитория** или сайта на Netlify;
- **удаление** файлов, веток, сайтов, репозиториев;
- всё **платное** и всё **необратимое**.

Правило простое: наружу (в интернет, в main, в деньги) — только после явного «да».

## Ветки и PR (когда проект растёт)

- Мелкая правка (текст, стиль, один файл) — можно коммитить в main напрямую, с согласия человека.
- Крупное (новая страница, форма, платёж, много файлов) — отдельная ветка → push →
  Pull Request (`gh pr create`) → показать человеку, что изменится → мердж только после его «да».
- Никогда не делай force-push в main.

## Если что-то сломалось

- Ошибка деплоя — прочитай лог (`netlify deploy` его показывает), объясни простыми словами,
  предложи ОДИН следующий шаг.
- Сайт «испортился» после деплоя — успокой: Git помнит всё. Можно откатиться
  (`git revert`) и задеплоить предыдущую версию.
- Не гадай молча: если не хватает данных (аккаунта, доступа, решения) — спроси.
